## SAST with SonarQube

> Static analysis, code quality gates, SonarQube + Jenkins integration

---

### Theory

#### What is SAST (Static Application Security Testing)?

Static Application Security Testing analyzes source code, bytecode, or binary code **without executing the application**. It scans for vulnerabilities, coding errors, and quality issues early in the development lifecycle — often called a "shift-left" security approach.

Key characteristics:
- Runs against source code (no running app needed)
- Integrated into CI/CD pipelines
- Fast feedback loop for developers
- Catches issues before code reaches production

---

#### SonarQube Core Concepts

| Concept | Description |
|---|---|
| **Rules** | Individual checks applied to code (e.g., "avoid SQL concatenation") |
| **Issues** | Violations of rules found in the scanned code |
| **Code Smells** | Maintainability problems that don't break functionality but increase technical debt |
| **Bugs** | Code that is likely to behave incorrectly at runtime |
| **Vulnerabilities** | Security weaknesses that could be exploited |
| **Security Hotspots** | Code that requires manual review to determine if it's actually a vulnerability |
| **Technical Debt** | Estimated time to fix all code smells |
| **Coverage** | Percentage of code exercised by unit tests |

---

#### Quality Gate

A **Quality Gate** is a set of conditions that must pass for a build to be considered healthy. If any condition fails, the pipeline fails.

Common Quality Gate conditions:
- Code coverage must be ≥ 70%
- No new critical or blocker vulnerabilities
- Duplicated lines < 3%
- Maintainability rating ≥ A
- Reliability rating ≥ A

```
Build → SonarQube Scan → Quality Gate Check → Pass ✅ / Fail ❌
```

---

#### sonar-scanner Configuration

The scanner is configured via `sonar-project.properties` in the project root:

```properties
# sonar-project.properties

sonar.projectKey=my-nodejs-app
sonar.projectName=My Node.js Application
sonar.projectVersion=1.0

# Source directories
sonar.sources=src
sonar.tests=tests

# Language
sonar.language=js

# Coverage report (generated by Jest/Istanbul)
sonar.javascript.lcov.reportPaths=coverage/lcov.info

# Exclusions
sonar.exclusions=node_modules/**,coverage/**

# SonarQube server
sonar.host.url=http://<sonarqube-host>:9000
sonar.login=<your-token>
```

Run the scanner:

```bash
sonar-scanner
```

Or via Maven/Gradle plugin, or directly in a Jenkinsfile stage.

---

#### OWASP Top 10 and How SAST Catches Them

| OWASP Risk | How SAST Detects It |
|---|---|
| **A01 - Broken Access Control** | Flags missing authorization checks on routes/functions |
| **A02 - Cryptographic Failures** | Detects weak algorithms (MD5, SHA1), hardcoded keys |
| **A03 - Injection** | Identifies SQL/command/LDAP string concatenation |
| **A04 - Insecure Design** | Code smell rules flag unsafe design patterns |
| **A05 - Security Misconfiguration** | Flags debug modes, default credentials in code |
| **A06 - Vulnerable Components** | Dependency scanning (combined with SCA tools) |
| **A07 - Auth Failures** | Detects missing session invalidation, weak password logic |
| **A08 - Data Integrity Failures** | Flags unsigned data deserialization patterns |
| **A09 - Logging Failures** | Detects logging of sensitive data (passwords, tokens) |
| **A10 - SSRF** | Flags unvalidated user-controlled URLs in HTTP calls |

#### 2. Configure SonarQube in Jenkins + Quality Gate Stage

In Jenkins, go to **Manage Jenkins → Configure System → SonarQube Servers** and add:
- Name: `SonarQube`
- URL: `http://<sonarqube-host>:9000`
- Token: (generate from SonarQube → My Account → Security)

**Jenkinsfile with SonarQube + Quality Gate:**

```groovy
pipeline {
    agent any

    tools {
        nodejs 'NodeJS-18'
    }

    stages {
        stage('Install') {
            steps {
                sh 'npm install'
            }
        }

        stage('Test & Coverage') {
            steps {
                sh 'npm test -- --coverage'
            }
        }

        stage('SonarQube Analysis') {
            steps {
                withSonarQubeEnv('SonarQube') {
                    sh '''
                        sonar-scanner \
                          -Dsonar.projectKey=my-nodejs-app \
                          -Dsonar.sources=src \
                          -Dsonar.tests=tests \
                          -Dsonar.javascript.lcov.reportPaths=coverage/lcov.info
                    '''
                }
            }
        }

        stage('Quality Gate') {
            steps {
                timeout(time: 5, unit: 'MINUTES') {
                    waitForQualityGate abortPipeline: true
                }
            }
        }

        stage('Deploy') {
            when {
                expression { currentBuild.result == null }
            }
            steps {
                sh 'echo "Deploying application..."'
            }
        }
    }
}
```

#### 3. Run Scan on Node.js App & Fix Reported Issues

After the first scan, SonarQube will show issues in the dashboard. Common fixes:

**Issue 1 — SQL Injection (Vulnerability)**
```javascript
// ❌ Before (vulnerable)
const query = "SELECT * FROM users WHERE id = " + req.params.id;
db.query(query);

// ✅ After (fixed)
const query = "SELECT * FROM users WHERE id = ?";
db.query(query, [req.params.id]);
```

**Issue 2 — Hardcoded Credentials (Security Hotspot)**
```javascript
// ❌ Before
const dbPassword = "supersecret123";

// ✅ After
const dbPassword = process.env.DB_PASSWORD;
```

### Assignment 

> Set a Quality Gate that fails the build if **coverage < 70%** or **critical vulnerabilities exist**.

**Steps:**
1. In SonarQube, go to **Quality Gates → Create** (name it `Strict Gate`)
2. Add condition: `Coverage` is less than `70` → on **New Code**
3. Add condition: `Vulnerabilities` is greater than `0` with severity `Critical`
4. Go to your project → **Project Settings → Quality Gate** → assign `Strict Gate`
5. Re-run the Jenkins pipeline and verify it fails when conditions are not met

## DAST with OWASP ZAP

> Dynamic analysis, ZAP baseline scan, integrate into Jenkins

---

### Theory

#### SAST vs DAST

| Aspect | SAST | DAST |
|---|---|---|
| **When it runs** | Before deployment (source code) | After deployment (running app) |
| **What it needs** | Source code | A running application URL |
| **What it finds** | Code-level vulnerabilities | Runtime/behavioral vulnerabilities |
| **False positives** | Higher (no runtime context) | Lower (real HTTP traffic) |
| **Pipeline stage** | Build / test stage | After deploy-to-test stage |
| **Examples** | SonarQube, Semgrep, Checkmarx | OWASP ZAP, Burp Suite |

Use **both** — SAST for early feedback, DAST for runtime validation.

---

#### OWASP ZAP Modes

| Mode | Description | Use Case |
|---|---|---|
| **Baseline Scan** | Passive scan only — no active attacks | Quick check in CI/CD, safe for production-like envs |
| **Full Scan** | Passive + active attacks | Thorough testing in isolated test environments |
| **API Scan** | Targets REST/GraphQL/SOAP APIs using an OpenAPI/Swagger spec | API-focused pipelines |

```bash
# Baseline scan (passive only)
docker run -t zaproxy/zap-stable zap-baseline.py -t http://target-app

# Full scan (active attacks)
docker run -t zaproxy/zap-stable zap-full-scan.py -t http://target-app

# API scan with OpenAPI spec
docker run -t zaproxy/zap-stable zap-api-scan.py \
  -t http://target-app/api/openapi.json -f openapi
```

---

#### ZAP Docker Image Usage in Pipelines

The official ZAP Docker images make it easy to integrate into any CI/CD system:

```bash
# Pull the stable image
docker pull zaproxy/zap-stable

# Run baseline scan and output reports
docker run --rm \
  -v $(pwd)/zap-reports:/zap/wrk \
  zaproxy/zap-stable zap-baseline.py \
    -t http://target-app:3000 \
    -r zap-report.html \
    -J zap-report.json \
    -x zap-report.xml
```

Reports are saved to `./zap-reports/` on the host.

---

#### Parsing ZAP JSON Reports & Failing the Pipeline

ZAP reports alerts at four risk levels: `High`, `Medium`, `Low`, `Informational`.

```bash
#!/bin/bash
# parse-zap.sh — fail pipeline if any FAIL-level alerts exist

REPORT="zap-reports/zap-report.json"

HIGH_COUNT=$(jq '[.site[].alerts[] | select(.riskcode == "3")] | length' "$REPORT")
MEDIUM_COUNT=$(jq '[.site[].alerts[] | select(.riskcode == "2")] | length' "$REPORT")

echo "High risk alerts: $HIGH_COUNT"
echo "Medium risk alerts: $MEDIUM_COUNT"

if [ "$HIGH_COUNT" -gt 0 ]; then
  echo "❌ Pipeline FAILED: High-risk vulnerabilities found!"
  exit 1
fi

echo "✅ ZAP scan passed."
exit 0
```

---

#### OWASP Dependency-Check for Vulnerable npm Packages

The OWASP Dependency-Check tool scans project dependencies against the **NVD (National Vulnerability Database)** to find known CVEs.

```bash
# Run via Docker
docker run --rm \
  -v $(pwd):/src \
  -v $(pwd)/dc-reports:/report \
  owasp/dependency-check \
    --project "my-nodejs-app" \
    --scan /src \
    --format HTML \
    --format JSON \
    --out /report

# Or via npm (for Node.js projects)
npm audit
npm audit --json > npm-audit-report.json

# Fail on high severity
npm audit --audit-level=high
```

### Practical Lab

#### 1. Run OWASP ZAP Baseline Scan Against Node.js App

First, ensure your app container is running:

```bash
docker run -d \
  --name nodejs-app \
  --network app-network \
  -p 3000:3000 \
  my-nodejs-app:latest
```

Then run ZAP against it:

```bash
docker run --rm \
  --network app-network \
  -v $(pwd)/zap-reports:/zap/wrk \
  zaproxy/zap-stable zap-baseline.py \
    -t http://nodejs-app:3000 \
    -r zap-report.html \
    -J zap-report.json \
    -I   # Don't fail on warnings (WARN), only on FAIL alerts
```

---

#### 2. Add ZAP Stage to Jenkinsfile (After deploy-to-test)

```groovy
pipeline {
    agent any

    stages {
        stage('Build') {
            steps {
                sh 'docker build -t my-nodejs-app:latest .'
            }
        }

        stage('Deploy to Test') {
            steps {
                sh '''
                    docker network create app-network || true
                    docker rm -f nodejs-app || true
                    docker run -d \
                      --name nodejs-app \
                      --network app-network \
                      -p 3000:3000 \
                      my-nodejs-app:latest
                    sleep 10  # Wait for app to be ready
                '''
            }
        }

        stage('OWASP ZAP Baseline Scan') {
            steps {
                sh '''
                    mkdir -p zap-reports
                    docker run --rm \
                      --network app-network \
                      -v $(pwd)/zap-reports:/zap/wrk \
                      zaproxy/zap-stable zap-baseline.py \
                        -t http://nodejs-app:3000 \
                        -r zap-report.html \
                        -J zap-report.json \
                        -I
                '''
            }
            post {
                always {
                    // Archive reports regardless of scan result
                    archiveArtifacts artifacts: 'zap-reports/**', allowEmptyArchive: true
                    publishHTML([
                        allowMissing: false,
                        alwaysLinkToLastBuild: true,
                        keepAll: true,
                        reportDir: 'zap-reports',
                        reportFiles: 'zap-report.html',
                        reportName: 'OWASP ZAP Report'
                    ])
                }
            }
        }

        stage('Check ZAP Results') {
            steps {
                sh '''
                    HIGH=$(jq '[.site[].alerts[] | select(.riskcode == "3")] | length' zap-reports/zap-report.json)
                    echo "High risk alerts: $HIGH"
                    if [ "$HIGH" -gt 0 ]; then
                        echo "Pipeline failed due to high-risk vulnerabilities"
                        exit 1
                    fi
                '''
            }
        }
    }

    post {
        cleanup {
            sh 'docker rm -f nodejs-app || true'
        }
    }
}
```

#### 3. Generate HTML Report and Archive as Jenkins Artifact

The `archiveArtifacts` and `publishHTML` steps in the Jenkinsfile above handle this. After the build:
- Go to the Jenkins build page
- Click **OWASP ZAP Report** in the left sidebar to view the HTML report
- Click **Build Artifacts** to download the raw JSON/HTML files

### Assignment 

> Run `npm audit` and `owasp-dependency-check` on the Node.js project.

**Part 1 — npm audit:**

```bash
# Basic audit
npm audit

# Output JSON report
npm audit --json > reports/npm-audit.json

# Fail CI if high or critical vulnerabilities found
npm audit --audit-level=high

# Attempt auto-fix
npm audit fix
```

**Part 2 — OWASP Dependency-Check in Jenkins:**

```groovy
stage('Dependency Check') {
    steps {
        sh '''
            mkdir -p dc-reports
            docker run --rm \
              -v $(pwd):/src \
              -v $(pwd)/dc-reports:/report \
              owasp/dependency-check:latest \
                --project "my-nodejs-app" \
                --scan /src/package-lock.json \
                --format HTML \
                --format JSON \
                --out /report \
                --failOnCVSS 7
        '''
    }
    post {
        always {
            archiveArtifacts artifacts: 'dc-reports/**', allowEmptyArchive: true
            publishHTML([
                reportDir: 'dc-reports',
                reportFiles: 'dependency-check-report.html',
                reportName: 'OWASP Dependency Check Report'
            ])
        }
    }
}
```

`--failOnCVSS 7` causes the stage to fail if any dependency has a CVSS score of 7.0 or higher (High/Critical).

## Quick Reference Summary

```
SAST Pipeline Flow:
Code → Build → Unit Tests → SonarQube Scan → Quality Gate → Deploy

DAST Pipeline Flow:
Code → Build → Deploy to Test → ZAP Scan → Dependency Check → Deploy to Prod
```

| Tool | Type | Stage | Fails On |
|---|---|---|---|
| SonarQube | SAST | Pre-deploy | Quality Gate conditions |
| OWASP ZAP | DAST | Post-deploy (test env) | High/FAIL alerts |
| npm audit | SCA | Build | High/Critical CVEs |
| OWASP Dependency-Check | SCA | Build | CVSS ≥ 7.0 |
